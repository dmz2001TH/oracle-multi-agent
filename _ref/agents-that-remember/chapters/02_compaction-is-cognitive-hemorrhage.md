# Chapter 2: Compaction Is Cognitive Hemorrhage

Compaction is the silent event at the center of every long agent session. It happens automatically. It happens often. It is the single largest source of "the agent forgot" bug reports in every multi-turn framework I have worked with. And it is treated, in most documentation, as a minor footnote — "when the context fills, older turns are summarized" — instead of as what it actually is: **the planned destruction of specifics in exchange for continued operation**.

This chapter is not a polemic against compaction. We need it. A context window that refuses to compact dies when it fills, and a dead agent remembers nothing at all. Compaction is a survival mechanism. But survival mechanisms are triage, and triage bleeds. You have to plan for the bleeding.

I call it cognitive hemorrhage because the metaphor is accurate. Blood volume you can replace. Oxygen-deprived tissue you cannot. Compaction replaces volume — the count of tokens in the window stays roughly stable — while the specific, load-bearing details are exactly the things that do not survive the replacement. The agent continues to function. The particular decisions that made that function *correct* do not.

The goal of this chapter is to put numbers on the phenomenon so that later chapters can design around it without hand-waving.

---

## What compaction actually is

Different harnesses implement compaction differently, but the family of algorithms is narrow. Variations are one of:

1. **Rolling summarization**: when the window exceeds a threshold, the oldest N turns are replaced by a single summary turn generated by the same model (or a cheaper one). New turns continue to arrive. Eventually the summary itself gets re-summarized as more turns age out.

2. **Hierarchical summarization**: the transcript is kept in a tree — leaf turns, chunk summaries, chapter summaries — and as the window fills, leaves are pruned and the agent is shown progressively coarser summaries for older material.

3. **Relevance-filtered retention**: an embedding-based scoring pass decides which older turns "look relevant" to the current task and keeps those, discarding the rest. This sounds smart. It is, in my experience, the most dangerous of the three, because the selector is wrong often enough that the agent silently loses things the model will need three turns from now.

All three produce the same outcome from the agent's point of view: **old specifics get replaced by paraphrase**. The paraphrase is shorter. The paraphrase lacks the things that would let you reconstruct the original. And the agent is given no flag that a replacement happened.

That last part is what makes compaction a hemorrhage rather than a filing cabinet. A filing cabinet says "the document is in drawer B." Compaction says "there is no document. Here is what the document was about." The former is a pointer. The latter is a loss.

## The math of the 100-hour session

Let me put concrete numbers on this, because I think the quiet scandal of long-session work is that nobody has made the math obvious.

Assume a working agent in a typical coding session. I will use values that are on the conservative side of what I actually observe in the maw-js federation, because I would rather understate the problem than overstate it.

- **Context window**: 200K tokens. This is the standard for the models I use most. Some are larger. The qualitative picture does not change.
- **Typical file the agent reads**: 2–3K tokens. A single `.ts` file in our mailbox module is around 2.1K tokens. A dense config file can be 500. A generated migration can be 4K.
- **Typical tool output per turn**: 1–2K tokens. A `git log --oneline -20` is tight. A `git status` after a long-running feature branch is 1.5K. A failing test with a stack trace is 3K. A successful `pnpm build` with warnings can be 8K.
- **Net new tokens per substantive turn**: roughly **2K**. That is the figure I have been measuring for years and I trust it as a working estimate.

Now do the arithmetic.

- **Turns until the window fills at 2K net/turn**: 200,000 / 2,000 = **100 turns**.
- **Duration of 100 turns at a realistic 10 turns per hour** (which includes thinking time, tool latency, the user reading output): **10 hours**.
- **Duration of 100 turns at an aggressive 30 turns per hour** (pair-programming pace, short prompts, fast tools): **~3.5 hours**.

So: a dense single-day session hits compaction. A 100-hour session hits compaction **tens of times**. Depending on the harness, compaction is not a once-ever event — the agent's summary of turn 1 gets re-summarized when the summary itself ages out. Each re-summarization is another lossy compression pass. Information decays geometrically.

This is why I say compaction is not an edge case in long work. It is the dominant regime. Any architecture that assumes "usually the full transcript is in the window" is wrong after hour 10.

## What actually gets lost

Compaction is not a uniform blur. It has a shape, and you can predict what will go first if you know how these summarizers are tuned.

**First to go — always**:

- Exact file paths. "We edited the mailbox code" survives. "`maw-js/src/modules/mailbox/standing-orders.ts`" does not.
- Function and variable names. "We refactored the retry logic" survives. "We renamed `enqueueWithBackoff` to `scheduleRetry`" does not.
- Numeric specifics. "We tuned the timeout" survives. "We set it to 4200ms because 5000 tripped the CDN's idle-connection cutoff" does not.
- Decisions and their *reasons*. "We chose JSONB for the payload column" might survive. "We chose it over a normalized side table because payload shapes were expected to proliferate" rarely does.
- Things the user said that the agent agreed to but did not immediately act on. "The user asked us to handle the empty-inbox case" — gone by turn 40. This is a recurring failure mode in our mailbox work.

**Second to go**:

- Tool output particulars. The fact that there was a failing test on commit X is preserved as "tests were failing"; the specific assertion line disappears.
- Sequence order. Which refactor came before which. The agent remembers that both happened; it loses causality.
- Constraints the user imposed early ("don't touch the auth module") — these evaporate after enough turns and the agent happily touches the auth module.

**Fairly durable**:

- The high-level task the session is about. "We are building a mailbox system" is robust.
- Strong user preferences repeated several times. "The user prefers tests to be colocated" survives if they said it more than once.
- The agent's own persona and system-prompt-level instructions.

If you look at that list, what gets lost is exactly what an engineer needs to do correct work. What survives is the sort of thing a project manager would put in a status update. That is not a coincidence: compaction summarizers are trained on data where "summarize this conversation" produces a manager-friendly paraphrase, not a log of decisions with their justifications. We inherited the wrong summarization objective, and the tooling has not caught up.

## Guaranteed failure, not probable failure

I want to name something carefully.

People sometimes respond to compaction warnings by saying "well, in practice the agent *usually* recovers — it re-reads the file, notices its mistake, and corrects itself." That is true *in practice*, when the user is in the loop catching mistakes. It is also a dangerous thing to build an architecture on. The existence of a human corrector makes it feel like the system works. But consider the autonomous cases — background agents, long-running tasks, overnight work — where there is no corrector. In those cases the agent operates on the post-compaction summary with no verification. Errors propagate. Errors compound.

In our federation we run overnight batch sessions where agents merge pull requests, close stale issues, and write retrospective notes. An early version of that pipeline had no ritual for re-reading state before acting. The agents would begin the night with a full context, work through several hours, hit compaction, and then make decisions on the compacted summary. We had nights where an agent "merged" a PR that had in fact been abandoned two weeks earlier, because the summary of the earlier discussion said "PR is ready" and the agent did not re-check. It was ready. Past tense. The summary dropped the tense.

That kind of mistake is the exact mechanism of the Hour-57 Problem from Chapter 1, scaled to a task with no user in the loop. It is why we now wrap every batch action with a read-the-source ritual before acting — a subject Part II returns to in detail.

## Why "just use a bigger window" is not the answer (again, with math)

Chapter 1 sketched this; here is the numerical version.

Suppose context windows grow to 1M tokens tomorrow (some already have). Re-run the math:

- Turns until fill at 2K net/turn: **500**.
- At 10 turns per hour: **50 hours**.
- At 30 turns per hour: **~17 hours**.

So a 1M-token window pushes the first compaction event past the single-day mark for moderate-pace work. That is genuinely an improvement. It is not a solution for multi-day work, and it is worth noting that every team I know that has a 1M-token model available is *already* running into the same problems, because their work has expanded to fill the bigger room. Parkinson's Law applies to tokens.

Also, large windows have secondary failure modes that bear on our subject:

- **Attention dilution**. As context grows, the model's attention per token declines. A specific decision buried at token 400K is less influential on generation than the same decision at token 20K. The agent nominally "has" the information and functionally does not use it.
- **Read cost**. A 1M-token context takes longer to process per turn and costs more. Real deployments run cheaper tiers or truncate to stay affordable. Truncation is compaction under a different name.
- **Test coverage on models**. Most models are evaluated most heavily on short-context tasks. Long-context behavior is the tail of the distribution and has more surprises. Bugs hide there.

The empirical takeaway: bigger windows are nice. Do not build on them as though they eliminate the problem. Assume compaction will happen, and design around that assumption.

## How I think about it now

After enough cycles I stopped treating "the conversation so far" as the record of what had happened. I started treating it as a **cache** over the real record, which lives on disk. The transition sounds semantic but has operational consequences.

If the conversation is the record:

- The agent defers to its own memory when asked a question.
- Re-reading is a waste.
- The architecture is "trust the window."

If the conversation is a cache:

- The source of truth is the filesystem (or a database, or git, or a mailbox store — something durable).
- Re-reading is the default, especially before asserting or acting.
- The architecture is "write before you forget; re-read before you claim."

The cache framing has a nice consequence: it makes compaction a *planned* event rather than a surprise. Caches are supposed to get evicted. We know how to handle cache misses — you go to the source. That is exactly the habit this book is trying to install in agent architectures.

## Designing for guaranteed compaction

What do you actually do with this math? A few principles, expanded in later chapters but worth naming here so the design space is clear.

1. **Write durable state early and often.** Do not wait until "end of session" to save what you know. End-of-session may be after compaction and you will be saving a compressed echo, not the real thing. Write at decision points, when they are fresh.

2. **Keep the durable state small and structured.** If your "memory" is a giant free-text dump, you will end up paying the compaction tax on it anyway when you try to load it into the next session. Structure lets you load only the parts you need. In the maw-js vault, agent mailboxes are separate files per topic, not one long log.

3. **Make re-reading cheap.** If re-reading the source of truth costs 50K tokens, the agent will avoid it. If it costs 2K tokens because the source is tidy, the agent will re-read freely. Tidy matters operationally, not aesthetically.

4. **Ritualize the moments of vulnerability.** The moments where compaction matters most are: before making a claim, before making a non-trivial action, at handoff to another agent or to the next session. Those moments deserve explicit patterns, not "try to remember."

5. **Assume the agent will forget, and make it cheap to recover.** Good memory architecture is not about preventing forgetting — it is about making the cost of a forgotten fact low, because the real store is right there and the retrieval is scripted.

## A taxonomy of compaction events

Not all compaction events are equal, and the mitigation differs by type. After enough sessions I started labeling them.

**Scheduled compaction** happens when the harness detects the window crossing a threshold and proactively summarizes. This is the most benign case, because it is triggered by a deterministic condition you can reason about. You can anticipate it, write state just before it, and recover cleanly. Most long-session damage comes not from scheduled compaction but from the agent forgetting to write before it fires.

**Triggered compaction** happens mid-turn when a single tool output blows the budget — for example, an unexpectedly large file read or a verbose test log. The harness has to shed context to fit. These are nastier because they are not anticipated: the agent was in the middle of a thought, and suddenly earlier context is gone. Any state that was in flight but not yet acted on can be lost. The mitigation is to treat large tool outputs as external documents you never embed in the transcript — summarize them at read time, or route them through a separate agent, or write them to disk and refer to them by path.

**Cascading compaction** happens when an already-compacted summary gets re-summarized as even older material ages out. Each pass loses more. By the third or fourth cascade, the summary is so abstract it is indistinguishable from the system prompt's boilerplate. This is where "we discussed the mailbox earlier" becomes the final ghost of what used to be a detailed design decision. There is no clever in-window mitigation for cascading compaction. The only remedy is to have written durable state long before the cascade started, and to treat the in-window summary as decorative rather than informative once it has been compacted twice.

**Cross-agent compaction**, in multi-agent settings, is the case where agent A summarizes its conversation with agent B, and agent B summarizes the same conversation independently, and the two summaries diverge. Both agents now believe they had different conversations. This is a failure mode specific to federated systems, and it is the reason our mailbox protocol writes every cross-agent exchange to a durable shared log rather than trusting each agent's in-window recollection.

Each kind of compaction wants slightly different architectural treatment, but the common throughline is: **move the load-bearing bits out of the transcript and onto disk, before the compaction happens**.

## The specific shape of our mitigation

In maw-js we have a pattern called a **retrospective** — a file written at the end of each working session that records, in structured form: what we did, what we decided, what we tried and abandoned, what the next session should pick up. Retrospectives live in `ψ/memory/retrospectives/YYYY-MM/DD/HH.MM_topic.md`. They are small (typically under 400 lines), linked from an index, and written while the decisions are still in the uncompacted window.

The mailbox, the retrospectives, the per-agent standing-orders files, the session ledger — all of it is one strategy: **move the load-bearing specifics out of the context window and onto disk, before the window compacts them**.

The specifics of those patterns are Parts II and III of this book. This chapter is the why. The chapter after this dismantles the wrong answer — the "just use a vector database" reflex — so that when we get to the patterns, we are not re-litigating it.

## Takeaways

- **Compaction is lossy, silent, and guaranteed in long sessions.** Not an edge case.
- **At 2K net tokens per substantive turn, a 200K window fills in ~100 turns.** A 1M window in ~500. Do the arithmetic for your own session length.
- **What compaction destroys is exactly what engineering needs.** File paths, names, numbers, reasons, constraints.
- **Bigger windows postpone the problem; they also have attention-dilution and cost failure modes of their own.** Do not build on "we'll just have more tokens tomorrow."
- **Treat the conversation as a cache, not a record.** The durable store lives on disk.
- **Design with compaction as a planned event.** Write early, keep state structured, make re-reading cheap, ritualize the moments of vulnerability.
